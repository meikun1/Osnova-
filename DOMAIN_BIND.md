# Как привязать свой домен

Всего 3 шага. Всё остальное бот делает сам.

---

## Шаг 1. Купить домен

Зарегистрируйте любой домен у любого регистратора:
Namecheap, REG.RU, GoDaddy, Hostinger, Beget — любой, где вам удобно.

Подойдёт любая зона: `.com`, `.ru`, `.online`, `.xyz` и т.д.

> Никаких настроек DNS у регистратора пока трогать **не нужно** —
> это будет на шаге 3.

---

## Шаг 2. Отправить домен боту

1. В главном меню нажмите **🌐 Привязать свой домен**.
2. Отправьте боту доменное имя — **просто имя, без `http://` и без `www`**:

   ✅ правильно: `example.com`
   ❌ неправильно: `https://example.com`, `www.example.com`, `example.com/`

3. Подождите несколько секунд. Бот в ответ пришлёт **два NS-сервера**, например:

   ```
   isla.ns.cloudflare.com
   kirk.ns.cloudflare.com
   ```

   Эти два адреса понадобятся на следующем шаге — скопируйте их.

---

## Шаг 3. Прописать NS у регистратора

Зайдите в личный кабинет регистратора, где покупали домен.

1. Откройте свой домен → раздел называется по-разному:
   **NS-серверы / Name Servers / DNS-серверы / Управление DNS**.
2. Выберите вариант **«свои NS»** / **«Custom DNS»** / **«Использовать другие NS»**.
3. Удалите старые NS и вставьте те **два адреса, которые прислал бот**.
4. Сохраните.

Готово. Дальше ничего делать не нужно.

---

## Когда домен заработает

NS обновляются у провайдеров от **1 до 24 часов** (обычно ~1–2 часа).
Как только обновятся — бот автоматически выпустит SSL-сертификат, и
ваш сайт откроется по адресу `https://ваш-домен`.

Проверить, что NS уже сменились, можно тут:
<https://www.whatsmydns.net/> — введите свой домен, выберите `NS`,
если по миру отображаются адреса Cloudflare — всё применилось.

---

## Что может пойти не так

| Сообщение бота | Что значит | Что делать |
|---|---|---|
| ❌ Невалидный формат домена | Отправили с `https://` / `www` / слешем | Отправить чистое имя: `example.com` |
| ❌ Домен уже зарегистрирован в другом Cloudflare-аккаунте | Этот домен раньше кто-то уже привязывал | Использовать другой домен |
| ❌ Cloudflare API: … | Временный сбой | Подождать минуту и попробовать снова |

Если ничего не помогло — напишите в поддержку.

# Для администратора бота

Раздел **только для того, кто разворачивает бота**. Обычному пользователю
читать не нужно — для него всё кончается на шаге 3 выше.

## Где создаётся Cloudflare API Token

1. Зарегистрируйтесь или войдите: <https://dash.cloudflare.com/login>.
2. Правый верхний угол → иконка профиля → **My Profile**
   (прямая ссылка: <https://dash.cloudflare.com/profile/api-tokens>).
3. Вкладка **API Tokens** → кнопка **Create Token**.
4. Выберите шаблон **«Edit zone DNS»** → **Use template**.
5. Настройте права:
   - **Permissions**:
     - `Zone` · `DNS` · `Edit`
     - `Zone` · `Zone` · `Read`
   - **Zone Resources**: `Include` · `All zones`
   - **TTL**: можно оставить без срока.
6. **Continue to summary** → **Create Token**.
7. **Скопируйте показанное значение** — оно показывается **один раз**.
   Формат: `cfut_...` (новый) или 40-символьная строка (старый).

## Пул CF-аккаунтов

Один CF-аккаунт = один пользователь бота навсегда (все его домены
лежат в этом аккаунте). Аккаунты лежат в таблице `cf_accounts` основной БД,
выдача атомарная (`FOR UPDATE SKIP LOCKED` на Postgres / lock на SQLite) —
один и тот же аккаунт **не уйдёт двум юзерам**.

Создавать аккаунты массово (один токен с правами `Zone:DNS:Edit` +
`Zone:Zone:Read` → All zones — см. раздел выше про каждый из них).
Сложите все токены в JSON, по шаблону `cf_pool.example.json`:

```json
[
  {"email": "acc1@gmail.com", "api_token": "cfut_xxx", "label": "acc-001"},
  {"email": "acc2@gmail.com", "api_token": "cfut_yyy", "label": "acc-002"}
]
```

Положите файл на сервер как `cf_pool.json` (или укажите свой путь в
`CF_POOL_JSON_PATH`). При старте бот импортирует новые аккаунты в БД —
**дубли по `api_token` пропускаются**, так что файл можно просто
дополнять и перезапускать бота, ничего не теряется.

В логах при старте увидите:

```
cf_pool: импорт завершён. Добавлено: N. В пуле всего: M, свободно: K.
```

Когда «свободно» = 0 — все юзеры, обращающиеся за новой привязкой,
получат сообщение «свободных аккаунтов нет, напишите админу» (никаких
зон не создастся). Долейте новых аккаунтов в JSON и перезапустите бота.

## Остальные настройки

| Переменная | Что вписать |
|---|---|
| `DOMAIN_SERVER_IP` | Публичный IPv4 сервера, на который указывают домены |
| `DOMAIN_CADDYFILE` | Абсолютный путь к Caddyfile (напр. `/etc/caddy/Caddyfile`) |
| `DOMAIN_CADDY_EXE` | Путь к бинарю Caddy (напр. `/usr/local/bin/caddy`) |
| `DOMAIN_TARGET` | Куда reverse_proxy шлёт трафик (по умолчанию `127.0.0.1:8000`) |
| `CF_POOL_JSON_PATH` | Путь к JSON-пулу (по умолчанию `cf_pool.json` в cwd) |
| `DOMAIN_BIND_GUIDE_URL` | URL этой инструкции (необязательно) |

> `DOMAIN_CF_TOKEN` больше **не используется** — токен берётся из пула.

## Caddy

Нужен билд с DNS-модулем Cloudflare (для DNS-01 challenge):

```bash
go install github.com/caddyserver/xcaddy/cmd/xcaddy@latest
xcaddy build --with github.com/caddy-dns/cloudflare
```

Caddyfile нужен **минимальный** — без глобального `cert_issuer`. Бот
пишет per-domain блок с собственным `tls` issuer'ом и токеном того
аккаунта, в котором лежит зона:

```caddy
{
    email YOUR_EMAIL@gmail.com
}

:80 {
    redir https://{host}{uri} 301
}
```

Запуск:

```bash
caddy run --config /etc/caddy/Caddyfile
```

При каждой привязке бот дописывает блок вида:

```caddy
example.com {
    reverse_proxy 127.0.0.1:8000
    encode gzip zstd
    tls {
        dns cloudflare cfut_TOKEN_OF_THIS_USER
    }
}
```

Так что сертификат для каждой зоны выпускается **тем токеном, который
владеет именно этой зоной** — пул из N разных CF-аккаунтов работает без
конфликтов.
